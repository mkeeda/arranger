# PR Review Workflow Skill

このスキルは、GitHubのPull Request (PR) に対してコードレビューを行うためのワークフロー定義です。
ユーザーからPRレビューを依頼された場合、またはPRに対するフィードバックを求められた場合、以下の手順に従ってレビューを実施してください。

## ワークフロー

### 1. PRの情報を取得する
対象のPRに関する情報をGitHub MCPサーバー経由で収集します。
- **`mcp_github_pull_request_read` (method: `get`)**: PRのタイトル、説明文(Body)、作成者、ステータスを取得し、背景と目的を理解します。
- **`mcp_github_pull_request_read` (method: `get_diff`)**: PRの変更内容（Diff）を取得し、どのようなコード変更が行われたかを把握します。情報が多すぎる場合は適宜ファイルを絞るなどして確認してください。
- *(必要な場合)* **`mcp_github_pull_request_read` (method: `get_check_runs`)**: CI等のステータスを確認し、テストが通っているか確認します。

### 2. コードを分析・レビューする
取得したDiffをもとに、以下の観点でコードを精査します。
- **バグの有無**: 論理エラー、NullReference等の考慮漏れ、エッジケースへの対応ができているか。
- **プロジェクト規約の遵守**: 都度プロジェクト内の `.agents/rules/code-style-guide.md` 等のルールファイルを参照し、最新のコードスタイル規約に従っているか。
- **パフォーマンスと設計**: 非効率な処理はないか。Jetpack Compose などのフレームワークのベストプラクティスに従っているか。
- **public APIの一貫性**: publicなクラスやメソッドの設計がプロジェクト内で一貫しているか。
- **実装のKotlinらしさ**: KotlinのIdiomに沿った、簡潔で意図が伝わりやすい実装になっているか。
- **命名の統一性、ニュアンスの正しさ**: 変数や関数の名前が統一されており、英単語のニュアンスが文脈に合っているか。
- **テストの抜け漏れ、過剰なケースはないか**: 必要なテストコードが追加・更新されているか。また、不要または過剰なテストを作ってしまっていないか。

### 3. レビュー内容の整理と方針決定
指摘事項や改善案、または賞賛（LGTMなど）をまとめます。
- **サマリー**: PR全体に対する所感や重要な指摘事項。
- **インラインコメント対象**: 修正が必要な箇所について、具体的なファイルパス、行番号、修正案（コードスニペット）を整理します。

*(推奨)* そのままGitHubに投稿する前に、「このような観点でレビューし、X件の指摘（またはApprove）を提出しようと思いますが進めてよいですか？」とユーザーに提案し、**合意を得てから**次のステップへ進むと安全です。

### 4. GitHub上でレビューを提出する
ユーザーの承認を得たら、実際にPRへレビューを送信します。
送信するレビューは3の内容を **全て英語で** 記述します。

1. **Pending Reviewの作成**:
   複数行へのコメントがある場合、`mcp_github_pull_request_review_write` (method: `create`, eventパラメーターなし) を使用して、下書き(Pending)状態のレビューを作成します。
2. **インラインコメントの追加**:
   `mcp_github_add_comment_to_pending_review` を用いて、特定ファイルの特定行に対してコメントを追加します。
3. **レビューの送信**:
   `mcp_github_pull_request_review_write` (method: `submit_pending` または、コメントを追加しない場合は `create`) を使用してレビューを確定・送信します。指定する `event` は以下の通りです。
   - 大きな問題がある / 修正が必須の場合: `REQUEST_CHANGES`
   - 問題がなく承認する場合: `APPROVE`
   - ちょっとした指摘やコメントのみの場合: `COMMENT`

## 指示事項
- レビューコメントは建設的で、理由とともに具体的な修正案を提示するように心がけてください。
- ユーザー自身のアカウントからAPI経由で投稿されるため、AIが実施したレビューであることが明確に伝わるよう、全体サマリー（本文の先頭）に `(AI review)` のプレフィックスを必ず追加してください。
- ユーザー自身のPRをレビューする場合、GitHubの仕様で `APPROVE` や `REQUEST_CHANGES` を行えないため、レビューの確定時は常に `event: COMMENT` を使用してください。
